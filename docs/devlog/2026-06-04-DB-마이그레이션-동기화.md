# DB ↔ 마이그레이션 SQL 동기화 일지

**날짜:** 2026-06-04
**맥락:** Phase 2 완료 직후, 라이브 DB와 마이그레이션 SQL 파일 간 불일치 정리

## 발견된 불일치

Phase 2 진행 중 라이브 DB에 다음 변경사항들이 직접 적용됐으나, 마이그레이션 SQL 파일에는 반영되지 않은 상태였음:

1. `regions.si` CHECK 제약에 `'all'` 값 허용 추가
2. `regions` 테이블에 3개 시 단위 시드 행 추가 (제주시/서귀포시/제주 전체)
3. `items.region_memo` 컬럼 추가
4. `items.item_name` 컬럼 추가
5. `item_categories` 조인 테이블 생성 + RLS 정책 3개 (SELECT/INSERT/DELETE)
6. `item_images` 테이블 생성 + RLS 정책 3개 (이미 01/02에 존재 — 멱등성 위해 05에 재선언)

## 진단 방법

Supabase SQL Editor에서 라이브 DB 실측:
- `information_schema.columns` — 컬럼 구조
- `information_schema.tables` — 테이블 목록
- `pg_constraint` — CHECK 제약
- `pg_policies` — RLS 정책 + qual/with_check
- `information_schema.referential_constraints` — FK + ON DELETE 동작

그 결과 라이브 DB의 정확한 상태를 파악하고, 그것을 재현하는 SQL 파일 작성.

## 해결: `supabase/migrations/05_phase2_adjustments.sql` 추가

모든 변경사항을 멱등성 보장 패턴 (`IF NOT EXISTS`, `ON CONFLICT DO NOTHING`, `DROP POLICY IF EXISTS` 등)으로 작성. 이미 적용된 상태에서 실행해도 에러 없이 통과.

새 환경 셋업 시 01 → 02 → 03 → 04 → 05 순서로 실행하면 라이브 DB와 동일한 상태가 됨.

## 검증

라이브 DB에서 05 파일 통째로 실행 → "Success. No rows returned" 통과 확인. 데이터 변경 없음, 에러 없음. 멱등성 보장됨.

## 교훈

1. **DB 변경 시 SQL 파일도 같이 업데이트하는 습관 필요**
   - 라이브 DB에 직접 변경하는 건 빠르지만 SQL 파일 업데이트 잊기 쉬움
   - 한 달 후엔 "왜 이렇게 됐지?" 추적 어려워짐

2. **포렌식 분석 방법 학습**
   - `information_schema`, `pg_*` 시스템 카탈로그로 라이브 DB 실측
   - 변경 이력이 없어도 현재 상태를 정확히 파악할 수 있음

3. **멱등성 보장 패턴의 가치**
   - `IF NOT EXISTS`, `ON CONFLICT DO NOTHING`, `DROP POLICY IF EXISTS` 등
   - "이미 적용된 환경에 다시 실행해도 안전"
   - 동기화/배포 안전성의 기본

4. **PostgreSQL 트랜잭션의 안전망**
   - 잘못된 SQL 실행 시도 시 트랜잭션 롤백으로 데이터 보호
   - "최악도 에러 메시지 뜨고 끝" — 부분 손상 없음
