# 2026-05-30 · 카카오 로그인 인프라 셋업

> Phase 1 (카카오 로그인) 코드 작성 전 준비 단계.
> 이 날은 코드는 짜지 않고 외부 서비스 셋업에만 집중함.

---

## 결과 한 줄

**카카오 로그인 → Supabase 세션 발급 → `public.profiles` 자동 생성**까지의 인프라 전 구간이 실제 테스트에 통과했다. 이제 짜투 앱 안에 "카카오로 시작하기" 버튼만 붙이면 로그인이 동작한다.

---

## 한 일

### 1. 카카오 개발자센터 (developers.kakao.com)

- **앱 등록** — 짜투 앱 생성 (App ID `1472661`)
- **앱 로고** — ZZATU 로고 업로드
- **카카오 로그인 활성화**
- **OpenID Connect(OIDC) 활성화** — Supabase 권장 방식
- **리다이렉트 URI 등록** — REST API 키 쪽, JavaScript 키 쪽 양쪽에 등록
- **동의 항목 설정**
  - 닉네임: **필수**
  - 프로필 사진: 선택
  - 카카오계정(이메일): 선택
- **클라이언트 시크릿 발급** → 한 번 노출되어 즉시 **재발급** 완료
- **카카오 비즈니스 인증** 완료 (개인사업자로 전환). 이메일 항목 권한이 풀림.

### 2. Supabase 측

- **Authentication → Providers → Kakao** 활성화
- 카카오에서 받은 **Client ID / Client Secret** 입력
- **Allow users without email** 토글 ON — 이메일 권한이 풀렸지만 일부 사용자가 이메일 미동의로 들어와도 가입은 가능하도록 안전장치
- (이미 셋업되어 있던 `handle_new_user` 트리거가 자동으로 `auth.users` → `public.profiles` 복사를 처리)

### 3. End-to-End 테스트

- 실제 카카오 계정으로 로그인 시도
- **결과**: 카카오 동의 화면 통과 → Supabase 세션 발급 → `auth.users`에 row 생성 → 트리거가 `public.profiles`에도 row 생성. 닉네임·이메일이 정상 입력됨.

## 새로 알게 된 것

### OAuth `state` 만료

OAuth 흐름에서 발급되는 `state` 값에는 짧은 만료 시간이 있다.
카카오 동의 화면에서 너무 오래 머무르면(예: 다른 일 하다가 늦게 클릭) `state`가 만료되어 회원 생성이 실패한다.

→ **함의**: 사용자가 로그인 도중 이탈했다 돌아오는 케이스는 "다시 처음부터" 안내해야 함. 에러 메시지·복귀 UX 설계 시 반영.

### `handle_new_user` 트리거 실제 동작 확인

스키마 셋업 단계에서 미리 만들어둔 PostgreSQL 트리거가 의도대로 동작하는 것을 확인.
`auth.users`에 행이 추가되는 순간 `public.profiles`에 같은 `id`로 행이 자동 생성된다.

→ **함의**: 회원가입 시 우리 앱 코드가 별도로 `profiles` insert를 호출할 필요 **없음**. 클라이언트 측에서는 "로그인 = 가입"으로 단순화 가능.

### 메타데이터 흐름

카카오 동의항목 → Supabase user metadata(`raw_user_meta_data`) → 트리거를 거쳐 `public.profiles`의 `nickname`/`email` 컬럼에 자동 입력된다.

→ **함의**: 카카오에서 받아온 닉네임이 그대로 짜투의 첫 닉네임이 됨. 사용자가 닉네임을 수정하고 싶으면 별도 "프로필 수정" 화면이 필요.
